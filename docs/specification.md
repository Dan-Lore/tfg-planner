# Спецификация продукта

> **Статус:** черновик v0.4.1 · живой документ  
> **Последнее обновление:** 2026-06-17

## 1. Видение

Веб-сайт для проектирования **производственных мнемосхем** под модпак [TerraFirmaGreg-Modern](https://github.com/TerraFirmaGreg-Team/Modpack-Modern).

Пользователь визуально собирает линии из механизмов, настраивает машины вручную, видит потоки материалов и может масштабировать производство. Данные рецептов соответствуют конкретной версии модпака и **извлекаются парсером** из репозитория модпака.

**Референс по UX расчётов:** [Factorio Calculator](https://kirkmcdonald.github.io/) (см. [inspiration.md](inspiration.md)).

**Недоделки без заглушек в коде:** [kanban.md](kanban.md).

## 3. Функциональные требования

### 3.1. Парсер и загрузка версий модпака

- **Парсер Modpack-Modern** — обязательный компонент; извлекает pack data из [репозитория TFG](https://github.com/TerraFirmaGreg-Team/Modpack-Modern) (KubeJS, конфиги, lock-файл). Детали: [parser.md](parser.md).
- Отдельное **меню «Версии»** (не смешивать с редактором схем).
- Пользователь выбирает версию модпака (например `0.12.8`).
- Приложение подгружает **нормализованный JSON-бандл** для этой версии.

**Критерии приёмки:**

- [ ] CLI/CI собирает pack data для выбранного тега релиза.
- [ ] (P2) Периодическое обновление и мониторинг версий — только через CI + статический deploy; см. [kanban.md](kanban.md) K-009.
- [ ] Список доступных версий отображается в меню.
- [ ] После выбора версии редактор использует только её данные.
- [ ] При смене версии — предупреждение, если схема несовместима.
- [ ] Отчёт сборки (`build-report.json`) фиксирует непокрытые скрипты без silent-fail.

### 3.2. Редактор мнемосхем

- **Лист (холст):** свободное размещение узлов-механизмов.
- **Узел** — тип машины + выбранный рецепт + **ручные** параметры (overclock, parallel, …).
- **Связь** — направленное соединение; передаёт поток предметов/жидкостей.
- Составление **производственных линий** из последовательных и параллельных участков.
- **История действий:** отмена и повтор операций на холсте.

**Горячие клавиши (MVP):**

| Действие | Windows / Linux | macOS (доп.) |
|----------|-----------------|--------------|
| Отменить | **Ctrl+Z** | Cmd+Z |
| Повторить | **Ctrl+Y** | Cmd+Shift+Z |

Отменяются: добавление/удаление узлов и связей, перемещение, дублирование, изменение рецепта и параметров машины, операции масштабирования (режим C), правки целевых скоростей.

**Критерии приёмки:**

- [ ] Добавление механизма из каталога версии.
- [ ] Соединение выхода одного узла со входом другого (валидация совместимости).
- [ ] Визуализация скоростей на связях после расчёта.
- [ ] На узле отображается `machineCount` (целое количество машин, ≥ 1).
- [ ] Ctrl+Z / Ctrl+Y работают в редакторе; стек undo не ломается при пересчёте потоков.

### 3.3. Калькулятор потоков (двусторонний, только продукты)

Расчёт касается **продуктов** (предметы, жидкости). Количество машин на узле — **целое число** (`machineCount`).

| Направление | Поведение |
|-------------|-----------|
| От цели | Пользователь задаёт «хочу N единиц/сек» на выходе → пересчёт потоков и **требуемых** `machineCount` по графу |
| От участка | Пользователь меняет скорость на связи / входе / выходе → **вся схема** пересчитывается согласованно |
| Топология | Новые узлы на холсте **не** создаются автоматически (кроме ручного режима A) |
| `machineCount` | Может задаваться вручную **или** обновляться калькулятором в режиме C после округления (§3.5.1) |

**Критерии приёмки:**

- [ ] Любое изменение целевой или промежуточной продуктовой скорости триггерит полный пересчёт.
- [ ] Потоки согласованы на всех рёбрах; byproducts учтены.
- [ ] `machineCount` на узлах всегда целое ≥ 1 после пересчёта.
- [ ] Unit-тесты на линейные, разветвлённые цепочки и дробные множители с округлением.

### 3.4. Энергия (EU/t)

- **Цель:** отображать энергопотребление на схеме (узел, линия, группа).
- **Модель pack data:** `Recipe.energy` = `EnergyStack` на **минимальном tier** рецепта: `{ minVoltageTier, voltage, amperage }`; `EU/t = voltage × amperage` на min tier; `amperage` **статичен** (не зависит от tier узла и overclock); `durationTicks` — время на min tier.
- **Узел схемы:** `voltageTier` (tier машины / energy hatch, ≥ min tier рецепта); `overclock` ускоряет рецепт (сокращает duration и повышает продуктовые потоки), **не** меняет EU/t. Multiblock energy hatch count / parallel — backlog **K-012** (не выводить из `amperage`).
- **GT voltage overclock:** при tier узла выше min tier — duration ÷ 2^Δtier, EU/t = `GTValues.V[nodeTier] × amperage` (amperage из pack).
- **Singleblock:** в pack data `amperage ≤ 1` на min tier; tier узла может быть выше min tier рецепта.
- **Запрещено:** показывать фиктивные значения (0, «—», placeholder), если данных нет.
- Энергия в pack data присутствует **только** когда парсер её достоверно извлёк.

**Критерии приёмки (по мере закрытия подзадач K-003):**

- [x] EU/t на узле — только при `recipe.energy` в pack data.
- [x] Tier picker на узле (≥ min tier); overclock не влияет на EU/t.
- [ ] Суммарное потребление линии / группы.
- [ ] Непокрытые рецепты остаются в kanban, не в UI (частично: ~75% без energy в 0.12.8).

### 3.5. Масштабирование линий

Два **независимых способа** увеличить производство. Пользователь может комбинировать их на одной схеме.

#### Режим A — ручное повторение линии

Пользователь **сам копирует** фрагмент схемы: дублирует узлы и связи на холсте (copy/paste или аналог), расставляет параллельные ветки, при необходимости объединяет потоки.

- Калькулятор пересчитывает потоки по фактической топологии.
- Автоматическое умножение узлов **не** выполняется.

#### Режим C — целевой продукт на выходе

Пользователь указывает **сколько продукта нужно в конце** линии (или на выбранном выходном узле): например «480 пластин меди / мин». Скорость может быть **дробной**.

- Калькулятор вычисляет требуемые `machineCount` по цепочке, **округляет** (§3.5.1), выполняет **полный пересчёт** потоков с целыми машинами.
- Новые узлы на холсте **не** появляются — меняется только `machineCount` на уже размещённых узлах (пользователь заранее выстраивает топологию).

#### 3.5.1. Дробные множители и округление машин

Целевые скорости (режим C) могут давать **дробное** «идеальное» число машин на узле.

**Инвариант:** после каждого пересчёта `machineCount ∈ ℤ`, `machineCount ≥ 1`.

**Пайплайн:**

```
дробный множитель / целевая скорость
        → идеальные machineCount (рациональные)
        → ceil(ideal) на каждом узле (min 1)
        → полный пересчёт продуктовых потоков с целыми машинами
```

**Округление:** всегда **вверх** (`ceil`), минимум 1 на узел.

```
идеальные machineCount (дробные)
        → ceil(ideal) на каждом узле
        → полный пересчёт продуктовых потоков
```

Гарантирует достаточную производительность; фактический выход может быть чуть выше целевого.

Пользователь видит на узле **фактическое** целое `machineCount` и итоговые потоки после пересчёта.

#### Общее для всех режимов

| Правило | Описание |
|---------|----------|
| Пересчёт | Любое масштабирование (C) и правка топологии (A) → полный пересчёт продуктовых потоков |
| Топология | Автодобавление/автоудаление **узлов** на холсте запрещено |
| `machineCount` | Обновляется вручную или калькулятором (C); всегда целое |
| Undo | Режим C попадает в стек **Ctrl+Z** / **Ctrl+Y** |

**Критерии приёмки:**

- [ ] Режим A: дублирование узлов/связей пользователем + корректный пересчёт потоков.
- [ ] Режим C: целевая скорость (дробной допускается) → округление `machineCount` → пересчёт.
- [ ] После любого режима: целые `machineCount`, согласованные потоки на рёбрах.
- [ ] Отмена/повтор масштабирования через Ctrl+Z / Ctrl+Y.

- Экспорт на диск / импорт с диска.
- Round-trip без потери графа и метаданных.

**Критерии приёмки:**

- [ ] Сохранить → загрузить → схема идентична.
- [ ] Предупреждение при несовпадении версии модпака.
