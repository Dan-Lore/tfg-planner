# Подписи рёбер и загрузка портов

> Дополнение к [specification.md](./specification.md). Зафиксировано 2026-06-23.

## Подписи на рёбрах

Каждое ребро в solver хранит **свой** `edgeFlows[id]`. Подписи на канвасе — только агрегированное отображение; потоки по отдельным рёбрам не сливаются.

### Source (у источника)

- **Одна подпись на физический выход** `out_i` узла.
- Значение = **сумма** `edgeFlows` всех рёбер с этим `(source, sourcePort)`.
- Fan-out на несколько целей, разные `itemId` на рёбрах и число рёбер **не** создают дополнительных source-подписей.
- Подпись рисуется на «центральном» ребре группы (ближе к centroid handle'ов источника).

**Пример:** теплица с `out_0`, `out_2`, `out_3`, каждый порт ведёт на две печи → **3** source-подписи (по сумме на порт), не 6.

### Target (у приёмника)

- **Одна подпись на физический вход** `in_j` узла.
- Значение = **сумма** `edgeFlows` всех рёбер с этим `(target, targetPort)`.
- Несколько источников / портов источника на один вход → одна target-подпись с полной суммой.
- Если все рёбра группы попадают в один handle, «центральное» ребро выбирается по позиции **sourcePort** (не target — координаты совпадают).

**Пример:** три ребра с разных выходов теплицы на `in_0` печи → **1** target-подпись ≈ сумма трёх потоков.

## Загрузка узла и портов

| Метрика | Формула | Где показывается |
|---------|---------|------------------|
| `nodePortInLoad[in_j]` | `inflow / demand` (cap 1) | % на входном порте |
| `nodePortOutLoad[out_i]` | `sent / produced` (cap 1) | % на выходном порте |
| `nodeLoad` | `min(min входов, min выходов)` | chip в шапке узла |

- `produced` — теоретическая скорость порта при текущем `machineCount`.
- `sent` — сумма `edgeFlows` по исходящим рёбрам с этого `out_i`.
- Неподключённые порты: load = 0%, tooltip «порт не подключён».

### Асимметрия одинаковых машин

Две одинаковые машины с одинаковым притоком на вход могут иметь **разную** общую загрузку и **разный** `%` на выходном порте, если downstream ограничивает забор (конкуренция за один вход следующего узла). Это ожидаемо и моделирует приоритет ближних связей в игре; solver **не** выравнивает потоки между симметричными конкурентами.
